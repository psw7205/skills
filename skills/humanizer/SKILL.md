# Humanizer

AI가 쓴 글처럼 보이는 표현을 검사하고, 필요하면 자연스럽게 다시 쓴다. 핵심은 의미 보존, 과잉 교정 방지, 작성자 목소리 유지다.

## Mode

- `검사`, `리뷰`, `확인`, `분석`, `AI 티 나?` 요청은 read-only로 처리한다. 원문 파일을 수정하지 말고 진단과 제안만 반환한다.
- `수정`, `고쳐`, `다듬어`, `rewrite`, `humanize`, `반영` 요청은 rewrite를 제공한다.
- 파일 경로가 주어졌더라도 사용자가 파일 반영을 명시하지 않으면 결과 텍스트만 제시한다.
- 글의 사실관계를 검증하는 스킬이 아니다. 근거가 필요한 주장이나 최신 정보가 보이면 스타일을 고치되 사실을 지어내지 말고 `unresolved: 근거 필요`로 남긴다.

## Reference

정식 검사나 rewrite를 할 때는 `references/patterns.md`를 읽고 패턴 목록을 적용한다. 입력이 한국어이거나 사용자가 `번역투`, `GPT 문체`, `한글 AI 티`를 언급하면 `references/korean-patterns.md`도 읽는다. 한두 문장짜리 가벼운 요청도 애매하면 reference를 확인한다.

## Workflow

1. **입력 파악**
   - 원문, 목적, 대상 독자, 유지해야 할 톤을 확인한다.
   - 사용자가 본인 글 샘플을 주면 먼저 읽고 문장 길이, 단어 수준, 문단 시작 방식, 구두점, 전환 습관을 메모한다.
   - 샘플이 없으면 자연스럽고 단정한 기본 톤을 쓴다. 기술 문서와 정책 문서는 중립적이고 plain한 톤이 사람다운 톤이다.

2. **패턴 진단**
   - 단일 징후가 아니라 클러스터를 본다. em dash 하나, curly quotes 하나, 잘 다듬어진 문장 하나만으로 AI라고 판단하지 않는다.
   - 과장된 의미 부여, 홍보 문체, vague attribution, present participle padding, rule of three, copula avoidance, generic conclusion, chatbot framing, diff-anchored writing을 우선 확인한다.
   - 한국어 글은 별도 fast path로 본다. 번역투, 영어식 피동, 형식명사, 문두 접속사, 연결어미 뒤 쉼표, 동일 종결어미 반복, "그/그녀/그것" 직역, 과한 불릿/볼드/이모지를 우선 스캔한다.

3. **Rewrite**
   - 원문이 다룬 내용을 빠뜨리지 않는다. 기본적으로 문단 수와 정보 범위를 유지한다.
   - 삭제보다 대체를 우선한다. 부풀린 문장은 구체적 주장으로, 모호한 주어는 실제 행위자로, 장식적 표현은 평이한 동사로 바꾼다.
   - 자료가 없는 구체 사례, 사람 이름, 수치, 연구명은 만들지 않는다.
   - 한국어 rewrite는 D(관용구) → A(번역투) → I(형식명사) → G/H(hedging·접속사) → F(수식) → B(영어 용어) → C/J(구조·장식) → E(리듬) 순서로 처리한다.
   - 한국어 원문이 격식체면 격식체를 유지한다. AI 티는 격식 자체가 아니라 번역투, 리듬, 과한 수사, 구조적 자동화다.
   - 최종 rewrite에는 em dash와 en dash를 남기지 않는다. `--`가 dash 역할이면 문장 구조를 바꾼다.

4. **Self-audit**
   - draft 작성 후 "왜 아직 AI처럼 보이는가?"를 스스로 묻고 남은 티를 짧게 적는다.
   - 리듬이 너무 균일한지, 대조 구조가 과한지, 마무리가 slogan처럼 들리는지, invented specificity가 없는지 확인한다.
   - 한국어 결과는 고유명사, 수치, 날짜, 단위, 직접 인용, 법률 조문, 영어 약어가 원문과 같은지 다시 본다. 변경률이 30%를 넘는다고 판단되면 과윤문 경고를 붙이고, 50%에 가까우면 보수적으로 되돌린다.
   - final rewrite를 다시 훑어 `—`, `–`, 과한 bold, emoji, inline-header list, title-case heading을 제거한다.

## Output

검사 모드:

```markdown
**진단**
- ...

**수정 제안**
- ...

**예시 수정**
...
```

Rewrite 모드:

```markdown
**초안**
...

**아직 AI처럼 보이는 점**
- ...

**최종**
...

**변경 요약**
- ...
```

사용자가 "최종 문장만", "copyable markdown", "diff만"처럼 형식을 지정하면 그 형식을 우선한다.

## Gotchas

- **과잉 humanizing**: 기술 문서, 정책 문서, reference text에 농담, 1인칭, 감정 표현을 넣으면 더 나빠진다. 해당 장르에서는 plain하고 정확한 문장이 목표다.
- **가짜 구체성**: AI 티를 줄이려고 출처 없는 수치, 인터뷰이, 연구명, 사례를 추가하지 않는다. 구체성이 없으면 문장을 좁히거나 `unresolved`로 남긴다.
- **한국어 직역 냄새**: 영어 humanizer 규칙을 그대로 적용하면 한국어 문장이 어색해진다. 한국어는 조사, 호응, 주어 생략, 높임, 문단 리듬을 함께 본다.
- **격식체 오판**: 공문, 리포트, 정책 문서의 딱딱한 문체를 전부 AI 티로 보지 않는다. 격식은 유지하고 번역투와 자동화된 표현만 줄인다.
- **과윤문**: "사람처럼"을 구어화나 문학화로 착각하지 않는다. 원문 장르가 리포트면 리포트답게, 블로그면 블로그답게 남긴다.
- **표식 하나로 단정**: em dash, curly quotes, title case, 완벽한 문법은 각각 단독으로 AI 증거가 아니다. 여러 패턴이 같이 나타날 때만 강하게 판단한다.
- **문서 종류 혼동**: changelog, release note, migration guide는 변경 이력을 쓰는 문서다. 일반 living document에서만 diff-anchored writing을 제거한다.
