# Plan

경량 plan 문서 작성과 리뷰를 한 스킬에서 제공한다.

## 철학

> 계획은 작업의 방향과 순서를 맞추기 위한 것이지, 구현을 선결정하기 위한 문서가 아니다.
>
> 리뷰는 이 계획이 완벽한가가 아니라, 지금 착수해도 큰 방향 오류가 없는가를 본다.

구현 디테일은 plan에 들어가지 않는다. 작업 중 plan 수정은 정상 동작이다.

## 모드 분기

사용자 발화로 모드를 결정한다. 모호하면 한 번 확인하고 진행.

- **Write 모드** — 새 plan 작성. "plan 써줘", "계획 세워줘" 류.
- **Review 모드** — 기존 plan 검토. "plan 리뷰해줘", "착수 가능해?" 류.

Write 모드는 저장 직후 **별도 서브에이전트로 Review 모드를 1회 자동 dispatch한다.** 같은 머리에서 작성과 비판을 동시에 하지 않는다는 원칙은 서브에이전트 위임으로 보존한다. 이 자동 리뷰는 끄지 않는다 — 결과는 사용자에게 그대로 전달되고, 후속 처리(수용·수정·무시)는 사용자 판단에 맡긴다.

## 공통 포맷

저장 경로: `docs/plans/YYYY-MM-DD-<topic>-plan.md`

### 헤더 (고정 문구)

모든 plan의 최상단에 다음 블록을 붙인다. Review 모드는 이 블록 유무로 plan 적합성을 1차 판단한다.

```
이 문서는 작업의 방향과 순서를 맞추기 위한 것이다.
구현을 선결정하지 않는다. 작업 중 plan 수정은 정상 동작이다.
```

### 섹션 (순서 고정)

1. **목표** — 1~2문장. 완료 상태를 정의.
2. **범위** — 포함 / 제외 항목.
3. **핵심 순서** — 3~7단계. 각 단계 = 제목 + 1~2줄 의도 + 검증 방법.
4. **리스크 & 선행검증** — 막힐 지점, 먼저 확인할 가정. 초반 단계에 배치 권장.
5. **열린 결정 (Open Items)** — 의도적으로 미정인 것. "구현 중 결정" 명시 OK. 비어 있어도 됨.

### 금지 항목 (plan 본문에 넣지 않음)

contract로 변해 구현 관성을 만들지 않도록 제거한다.

- 코드 블록 (언어 tagged fenced block)
- 함수 시그니처, 클래스 구조, 타입 정의
- 파일 경로 pin (`src/foo/bar.ts:L15` 같은 구체 참조)
- 에러 핸들링 디테일
- "확장 가능성", "향후 개선" 류 섹션

개념적 언급("설정 파일에 옵션을 추가", "컴포넌트 분리")은 OK. 결정을 박아두는 형태가 NO.

### 분량 가드

1~2페이지. 단계가 8개 이상이거나 페이지를 크게 초과하면 범위 분해를 제안한다. 이는 plan 분할이 아니라 **작업 범위 축소** 신호일 가능성이 크다.

## Write 모드

**입력**: 요구사항, spec 설명, 또는 대화 맥락.

### 절차

1. topic 결정 — 1~3단어 kebab-case. 애매하면 사용자에게 확인.
2. 5섹션 템플릿으로 초안 작성.
3. 저장 직전 자기 검증 (아래).
4. `docs/plans/YYYY-MM-DD-<topic>-plan.md`로 저장. 같은 날짜/토픽의 기존 파일이 있으면 덮어쓰기 전 확인.
5. **자동 Review dispatch** — Agent 도구로 `general-purpose` 서브에이전트를 1회 파견한다. 프롬프트에는 다음을 포함:
   - 저장한 plan 파일의 절대 경로
   - "Skill 도구로 `plan` 스킬을 Review 모드로 호출하여 이 plan을 검토하라"
   - 결론 한 줄(🟢/🟡/🔴) + finding 목록만 짧게 반환하라는 지시 (자체 분석 추가 금지)
   서브에이전트 결과를 사용자에게 그대로 전달하고 종료한다. 자동 재작성·자동 수정 없음.

### 자기 검증 (저장 전)

각 항목에 대해 점검하고 걸리면 수정:

- 헤더 고정 문구가 있는가?
- 본문에 코드 블록이 있는가? → 제거
- 파일 경로 pin이나 시그니처가 있는가? → 개념 언급으로 축약
- 각 단계에 검증 방법이 있는가? → 없으면 추가
- 단계 수가 3~7 밖인가? → 재분해하거나 범위 축소 제안
- "향후", "확장" 섹션이 있는가? → 제거

## Review 모드

**입력**: plan 파일 경로 또는 본문.

### 내부 프롬프트 (mental model)

```
이 plan이 완벽한가를 보지 않는다.
지금 착수해도 큰 방향 오류가 없는가를 본다.
구현 디테일 지적은 너의 일이 아니다.
```

### 볼 것 (Layer 1)

- 순서 뒤집힘 없는가
- 선행 의존성 누락 없는가
- 범위가 비정상적으로 크지 않은가
- 누락된 의사결정 포인트가 Open Items에 명시됐는가
- 되돌림 가능한가 (큰 irreversible 동작이 초반에 있는가)
- 큰 불확실성이 초반에 배치됐는가

### 보지 말 것

- 구현 세부 방식의 완결성
- 코드 구조의 미세한 선호
- 아직 결정할 필요 없는 세부안
- 먼 미래의 예외 케이스 / 확장 포인트

### Self-gate (finding마다 필수)

각 finding 후보를 리포트에 올리기 전에 아래를 통과시킨다:

```
"이게 Layer 1(방향·순서·범위·의존성·되돌림) 문제인가?"
  ├─ Yes → KEEP
  └─ No  → DROP (리포트에 적지 않음, 메타 언급도 금지)
```

DROP한 항목의 개수나 존재를 리포트에 언급하지 않는다. 사용자가 Layer 2를 원하면 brainstorming을 호출할 것.

### 출력 포맷

결론 하나로 닫는다:

- **🟢 진행 가능** — 1~2줄로 근거
- **🟡 조건부 진행** — 차단요소 N개, 각 한 줄
- **🔴 재정리 필요** — 흔들리는 전제 명시, 재정리 방향 1~2줄

### 상태

각 호출은 stateless. 카운트나 로그를 남기지 않는다. 반복 리뷰 제어는 사용자 책임.

## Gotchas

- plan 본문의 "코드 블록 금지"는 plan이 contract로 굳는 걸 막는 규칙. 이 SKILL.md 내부의 설명용 fenced block(헤더 템플릿, Self-gate 트리)과는 다른 층위.
- Review에서 "더 좋아질 수 있는가?"로 보면 종료 기준이 사라진다. 질문을 항상 "지금 착수 가능한가?"로 고정.
- Write에서 단계가 8개 이상이면 plan 분할이 아니라 작업 스코프 축소 신호인 경우가 많다. 먼저 범위 축소를 사용자와 논의.
- Review 모드에서 plan이 비어있거나 헤더 고정 문구가 없으면 진단 전에 "Write 모드로 먼저 작성 필요"를 안내.
- Open Items는 불완전의 표지가 아니라 "구현 시점에 결정"이라는 명시적 서명이다. Review에서 Open Items가 비어있다고 감점하지 말 것 — 진짜 미정이 없으면 그대로 두어야 한다.
- 자동 Review dispatch는 **Write 모드에서만 1회**다. 사용자가 Review 모드로 직접 호출했을 때는 또 dispatch하지 않는다 — 무한 위임 방지.
- 자동 Review 결과가 🔴여도 plan을 자동 재작성하지 않는다. 결과 보고로 끝내고 사용자가 다음 행동(수정 요청, 그대로 진행, 무시)을 결정한다.

## 범위 외

- Layer 2 (Design Notes, 구현 아이디어, 대안 비교) 작성 — brainstorming 또는 별도 design doc.
- 자동 실행·진행 관리 (execute-plan 류) — 이 스킬 담당 아님. `self-feedback-loop` 등과 조합.
