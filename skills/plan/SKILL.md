# Plan

계획은 작업의 방향과 순서를 맞추기 위한 것이지, 구현을 선결정하기 위한 문서가 아니다.
리뷰는 이 계획이 완벽한가가 아니라, 지금 착수해도 큰 방향 오류가 없는가를 본다.

구현 디테일은 plan에 들어가지 않는다. 작업 중 plan 수정은 정상 동작이다.

## 모드

- **Write** — "plan 써줘", "계획 세워줘" 류
- **Review** — "착수 가능해?", "plan 리뷰해줘" 류

모호하면 한 번 확인.

## Write 모드

저장 경로: `docs/plans/YYYY-MM-DD-<topic>-plan.md`. 같은 날짜·토픽 파일이 있으면 덮어쓰기 전 확인.

다음 요소를 작업 크기에 맞춰 자연스럽게 담는다 — 비어 있으면 생략 OK. 형식·분량·단계 수에 정답은 없다. 짧을수록 좋고, 1페이지를 넘기면 보통 작업 범위 축소 신호다.

- **목표** — 완료 상태 한 줄
- **범위** — 포함/제외가 모호할 때만
- **순서** — 단계와 의도. 결정적 단계(되돌리기 어렵거나 가정이 깨질 수 있는 곳)에만 검증 방법
- **리스크** — 막힐 가능성, 먼저 확인할 가정
- **열린 결정 (Open Items)** — 의도적으로 미정인 것. "구현 중 결정" 명시 OK. 진짜 없으면 비워둔다

### 절대 넣지 않는 것 (hard rule)

이게 plan을 contract로 굳히는 핵심 요인이라 이쪽만 강하게 막는다.

- 코드 블록 (언어 tagged fenced block)
- 함수 시그니처, 클래스 구조, 타입 정의
- 파일 경로 pin (`src/foo/bar.ts:L15` 같은 구체 참조)
- 에러 핸들링 디테일
- "향후 개선", "확장 가능성" 류 섹션

판정 기준: **개념 언급 OK / 결정 박는 형태 NO**. "설정 파일에 옵션 추가"는 OK, `appendOption(key: string)`은 NO.

### 저장 후 자동 Review

저장 직후 Agent 도구로 `general-purpose` 서브에이전트를 1회 dispatch한다. 이 dispatch는 생략하지 않는다 — 같은 머리로 작성·비판을 동시에 하지 않기 위함. 서브에이전트는 SKILL.md를 재로드하지 않으므로 아래 프롬프트만으로 동작한다.

서브에이전트 프롬프트 (그대로 또는 거의 그대로 전달):

```
너는 이 plan을 처음 본 독립 reviewer다.
작성자·대화 맥락 모름. plan 파일 본문만 읽고 답한다.
의심점만 출력한다 — 동의·격려·보강은 생략.

단일 질문: 지금 착수하면 어디서 막힐 가능성이 있는가?

볼 것: 순서 뒤집힘, 선행 의존성 누락, 범위 비대,
       되돌림 가능성, 큰 불확실성이 초반에 배치됐는지.
보지 말 것: 구현 디테일의 완결성, 코드 구조 선호,
            먼 미래 예외 케이스.

verdict 룰:
  의심점 0                  → 🟢 진행 가능 (근거 1줄)
  착수에 영향 주는 의심점 1+ → 🟡 조건부 (의심점 각 한 줄)
  전제 자체가 흔들림         → 🔴 재정리 필요 (흔들리는 전제 + 재정리 방향)

plan 파일 경로: <절대경로>
```

결과를 사용자에게 그대로 전달하고 종료. 🔴여도 자동 재작성 없음. 다음 행동(수정·진행·무시)은 사용자 판단.

## Review 모드

사용자가 직접 호출한 경우. plan 파일 경로를 받거나 `docs/plans/`에서 가장 최근 파일을 대상으로 한다.

위 서브에이전트 프롬프트와 같은 mental model + 같은 verdict 룰을 적용한다. 의심점만 출력하고 verdict 한 줄로 닫는다.

각 호출은 stateless. Review 모드에서는 자동 dispatch 없음 (무한 위임 방지).

## Gotchas

- "절대 넣지 않는 것"은 plan 본문 룰이지 SKILL.md 본문 룰이 아니다. 이 파일 안의 설명용 fenced block은 다른 층위.
- Open Items가 비어 있다고 감점하지 말 것. 진짜 미정이 없으면 그대로 두는 게 정답.
- 5분짜리 작업에 plan을 강요하지 말 것. 한두 줄 요약 + 즉시 착수가 나으면 그렇게 안내한다.
- 자동 Review 🟢이 곧 "이 plan이 좋다"는 뜻은 아니다. "착수 시 막힐 곳 안 보임"일 뿐.

## 범위 외

- Layer 2 (대안 비교, 구현 아이디어, design notes) — `brainstorming` 또는 별도 design doc.
- 자동 실행·진행 관리 — `self-feedback-loop` 등과 조합.
